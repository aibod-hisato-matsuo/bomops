## 📝 作業サマリー

将来の AIBOD 製品（RISC-Vボード等）を見据え、製品モデルを「ファミリ → グレード → バリエーション → モデル」の4層で分類できるようにし、部品マスタと同じカスケードUIを適用した。

**設計判断（過剰設計回避）:**
- UI 3案（カスケード踏襲/ツリーペイン/ファセット）からカスケードを推奨・採用（部品マスタとの操作一貫性・4層でも1行ずつしか増えない・疎データに強い）
- データモデルは「軽く始める」: **Family のみマスタ化**（`ProductFamily`・プラットフォーム構想の Domain Gateway 単位に対応・使用中削除PROTECT）、**grade/variation は nullable 自由文字列**（階層の切り方が固まったらマスタ昇格 — カテゴリで実証済みの移行パスを使う）。family FK も nullable にして既存データ・テストを壊さない（仮置きを明示）

**バックエンド:**
- マイグレーション0004: ProductFamily 作成＋grade/variation 追加＋データ移行（BAITEN STAND ファミリをシード、既存3モデルに family と grade=AI/Mini/不明 を自動設定）
- 新API: `/product-families/` CRUD、`/product-models/hierarchy-summary/`（family×grade×variation 件数集計）、ProductModelFilter（family名/grade/variation）
- admin に ProductFamily 管理、import_notion もファミリ/グレード付与に対応
- テスト3件新規（階層集計・複合フィルタ・PROTECT 409）。テストDBにはマイグレーションのシードが効くため get_or_create でフィクスチャ衝突を回避 — **47/47 パス**

**フロントエンド:**
- ピルボタン行を共通コンポーネント `CascadeRow` に切り出し（部品マスタと同デザイン・今後再利用可）
- 製品モデル一覧: ファミリ行（常時）→ グレード行（ファミリ選択時）→ バリエーション行（**値が実在する場合のみ出現**）→ 一覧。集計はクライアント側で階層ロールアップ。URL同期
- 一覧にファミリ/グレード/バリエーション列追加、編集フォームに「＋新規ファミリを追加…」（カテゴリと同じその場作成フロー）とグレード/バリエーション入力

**検証:** フロント23/23・lint/build クリーン。ローカルに RISC-V 試験データ3件（Pro/Standard×8GB/16GB）を投入し、BAITEN側はバリエーション行が出ない・RISC-V側は3層目まで展開・複合フィルタ（RISC-V×Pro×8GB→1件）を確認。CLAUDE.md 対応表更新。

**留意:** RISC-V 試験データはローカルのみ（本番には持ち込まれない）。未コミット・本番未デプロイ（確認待ち・本番反映時は migrate 0004）。
